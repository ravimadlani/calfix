Complete Guide to Publishing Your Google Calendar OAuth App

‚è±Ô∏è Expected Timeline: 2-4 Weeks

üìã Overview

To publish CalFix as a production app that anyone can use, you need to complete Google's OAuth verification process. Since CalFix requests https://www.googleapis.com/auth/calendar (full calendar access), this is considered a sensitive scope requiring enhanced verification.

üí° Why Verification is Required

Google requires verification for apps accessing sensitive user data (like calendars, email, or drive) to protect users from malicious apps. The process ensures your app follows security best practices and handles user data responsibly.

‚úÖ Prerequisites Checklist

Before starting the verification process, ensure you have:

  • Domain Name
    A registered domain for your app (e.g., calfix.com). Cannot use localhost or IP addresses.
  • Hosted Privacy Policy
    A publicly accessible privacy policy URL on your domain.
  • Hosted Terms of Service (Optional but recommended)
    Terms of service accessible on your domain.
  • App Homepage
    A landing page explaining what your app does.
  • Working Production App
    Your app must be deployed and functional at the domain you're verifying.
  • Video Demonstration
    A screen recording showing how your app uses Calendar data.
  • Domain Verification
    Proof of ownership for your domain in Google Search Console.

üìù Detailed Requirements

1Privacy Policy

Your privacy policy must be publicly accessible and cover:

  • What data you collect from Google Calendar
  • How you use the data
  • How you store and protect the data
  • Whether you share data with third parties
  • How users can delete their data
  • Contact information for privacy concerns

üîê Key Privacy Points for CalFix

Since CalFix processes calendar data entirely client-side:

  • ‚úì Data is NOT sent to your servers
  • ‚úì Data is NOT stored in databases
  • ‚úì OAuth tokens are stored in browser localStorage only
  • ‚úì All processing happens in the user's browser

This is a major security advantage you should highlight!

2Domain Verification

You must prove ownership of your domain:

  1. Go to Google Search Console
  2. Click "Add Property"
  3. Enter your domain (e.g., calfix.com)
  4. Verify ownership using one of these methods:
    • DNS TXT record (recommended)
    • HTML file upload
    • HTML meta tag
  5. Once verified, the domain appears in your OAuth consent screen settings
Google Search Console ‚Üí Domain Verification Guide ‚Üí

3Video Demonstration

Create a 2-5 minute screen recording showing:

  • OAuth Flow: Show the entire sign-in process
  • Permission Scopes: Clearly show what permissions you request
  • Data Usage: Demonstrate how you fetch and display calendar events
  • Features: Show all features that use calendar data (analytics, insights, actions)
  • Data Handling: Explain that data stays client-side (mention in narration)

üé• Video Recording Tips

  • Use screen recording software (QuickTime, Loom, or OBS)
  • Start from a logged-out state
  • Use a test Google account
  • Keep it concise but comprehensive
  • Upload to YouTube (unlisted) or Google Drive (public link)

4Authorized Domains

Update your OAuth consent screen with production domains:

  1. Go to OAuth Consent Screen
  2. Edit "Authorized domains"
  3. Add your domain (e.g., calfix.com)
  4. Update JavaScript origins and redirect URIs:
    • JavaScript origin: https://calfix.com
    • Redirect URI: https://calfix.com (or your OAuth callback URL)

‚ö†Ô∏è Important: HTTPS Required

Production apps MUST use HTTPS. Google will reject apps using HTTP or localhost URLs.

üìã Verification Questionnaire Preview

Google will ask questions like these. Prepare your answers in advance:

1. What does your application do?

Example answer: "CalFix is a calendar management dashboard that helps professionals and executive assistants optimize their Google Calendar. It provides analytics on meeting health, identifies scheduling issues (double bookings, back-to-back meetings, missing travel blocks), and offers one-click optimizations to improve calendar efficiency."

2. Why does your application need access to Google Calendar data?

Example answer: "CalFix requires full calendar access (read and write) to: (1) Fetch and analyze calendar events to calculate health scores and detect scheduling issues, (2) Create and modify events to implement optimizations like adding buffers, scheduling focus time, and creating travel blocks, (3) Access multiple calendars for users managing delegate calendars (e.g., executive assistants)."

3. How does your application use the data obtained from Google APIs?

Example answer: "CalFix processes all calendar data entirely within the user's browser (client-side). No calendar data is sent to our servers or stored in databases. The app analyzes events locally to provide insights and recommendations. Users can modify their calendars through the app, with all API calls going directly from the user's browser to Google's Calendar API."

4. How does your application store and protect user data?

Example answer: "CalFix uses a client-side architecture for maximum privacy. OAuth tokens are stored only in the user's browser localStorage. No calendar events or user data are transmitted to or stored on our servers. All data processing happens locally in the browser and is cleared when the user signs out."

5. Does your application share user data with third parties?

Example answer: "No. CalFix does not share, sell, or transmit user calendar data to any third parties. Since all processing is client-side, data never leaves the user's browser except for direct API calls to Google Calendar."

6. Provide a link to your privacy policy

Example: https://calfix.com/privacy

7. Provide a link to your terms of service

Example: https://calfix.com/terms

8. Provide a video demonstration of your app's OAuth flow

Example: https://youtu.be/your-video-id or https://drive.google.com/file/d/your-file-id/view

‚ö†Ô∏è Common Rejection Reasons & How to Avoid

1. Unclear Data Usage

Problem:

"Your application's use of calendar data is not clearly explained."

Solution:

  • Be very specific in your questionnaire answers
  • Show exactly how calendar data appears in your UI
  • Demonstrate every feature that uses calendar access

2. Poor Quality Video

Problem:

"Video demonstration does not clearly show OAuth flow or data usage."

Solution:

  • Record in HD (at least 1080p)
  • Show the entire OAuth consent screen
  • Demonstrate complete user journey
  • Keep video under 5 minutes but cover everything

3. Privacy Policy Issues

Problem:

"Privacy policy does not adequately describe data handling practices."

Solution:

  • Explicitly mention "Google Calendar data"
  • Describe client-side processing model
  • Include data retention and deletion policies
  • Provide contact information for privacy inquiries

4. Domain Not Verified

Problem:

"Authorized domain is not verified in Search Console."

Solution:

  • Verify domain BEFORE submitting for review
  • Ensure domain in OAuth settings matches verified domain
  • Use root domain, not subdomain (or verify both)

5. App Not Functional

Problem:

"Reviewers unable to access or use the application."

Solution:

  • Test production deployment thoroughly before submission
  • Ensure OAuth redirect URIs match production URLs
  • Provide test account credentials if app requires onboarding
  • Check that all links in submission work
